### 3. Database Models (Entities)

Here are all the main models you need:

| Model              | Key Fields |
|--------------------|----------|
| **User**           | `id`, `email`, `password` (hashed), `full_name`, `phone`, `role` (customer / restaurant_admin / rider / admin), `created_at`, `updated_at` |
| **Restaurant**     | `id`, `name`, `description`, `address`, `phone`, `image_url`, `rating`, `is_open`, `owner_id` (references User), `created_at` |
| **Category**       | `id`, `name`, `restaurant_id` |
| **MenuItem**       | `id`, `name`, `description`, `price`, `image_url`, `is_available`, `category_id`, `restaurant_id` |
| **Cart**           | `id`, `user_id`, `items` (JSON or separate CartItem table) |
| **Order**          | `id`, `user_id`, `restaurant_id`, `total_amount`, `status` (pending, confirmed, preparing, out_for_delivery, delivered, cancelled), `delivery_address`, `payment_status`, `created_at`, `updated_at` |
| **OrderItem**      | `id`, `order_id`, `menu_item_id`, `quantity`, `unit_price` |
| **Payment**        | `id`, `order_id`, `paystack_reference`, `amount`, `status`, `payment_method` |
| **Address**        | `id`, `user_id`, `street`, `city`, `state`, `coordinates` (lat, lng) |

**Note**: Start with `User`, `Restaurant`, `MenuItem`, `Order`, `OrderItem`. Add others later.

### 4. Architecture (Clean Architecture)

We will use this folder structure:

```
food-delivery-backend/
├── cmd/
│   └── app/
│       └── main.go
├── internal/
│   ├── config/          # Viper config loading
│   ├── model/           # All database structs (User.go, Order.go, etc.)
│   ├── dto/             # Request & Response structs (RegisterRequest, OrderResponse, etc.)
│   ├── repository/      # Database operations (UserRepository, OrderRepository)
│   ├── service/         # Business logic (UserService, OrderService, PaymentService)
│   ├── handler/         # Gin HTTP handlers (AuthHandler, OrderHandler)
│   ├── middleware/      # Auth, RateLimit, Logging, ErrorHandler
│   ├── queue/           # RabbitMQ producers & consumers
│   ├── util/            # JWT, ErrorResponse, PasswordHash, etc.
│   └── router/          # Route definitions
├── migrations/          # SQL or GORM auto migrations
├── pkg/                 # Shared utilities (optional)
├── docker-compose.yml
├── Dockerfile
├── .env.example
└── README.md
```

This structure is clean, testable, and looks very professional on GitHub.

### 5. Major Endpoints (API Routes)

**Auth**
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`

**Restaurants**
- `GET /api/v1/restaurants` (list with filters)
- `GET /api/v1/restaurants/:id`
- `POST /api/v1/restaurants` (restaurant admin only)

**Menu**
- `GET /api/v1/restaurants/:id/menu`
- `POST /api/v1/restaurants/:id/menu` (admin)

**Cart**
- `GET /api/v1/cart`
- `POST /api/v1/cart/items`
- `DELETE /api/v1/cart/items/:menu_item_id`

**Orders**
- `POST /api/v1/orders` (place order + initialize payment)
- `GET /api/v1/orders`
- `GET /api/v1/orders/:id`
- `PATCH /api/v1/orders/:id/status` (restaurant/rider)

**Payments**
- `POST /api/v1/payments/webhook` (Paystack webhook)
